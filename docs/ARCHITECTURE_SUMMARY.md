# Архитектурное резюме проекта Luminary

## 📋 Обзор проекта

**Luminary** — это AI-ревьюер кода для автоматического анализа Merge Requests в GitLab. Проект использует LLM (Large Language Models) для генерации конструктивных комментариев к коду, валидации этих комментариев и автоматической отправки их в GitLab.

**Текущий статус:** MVP завершен, этапы 1.1 и 1.2 выполнены, остальные этапы в планах. Проект функционален и готов к использованию.

**Версия:** 0.1.0

---

## 🏗️ Архитектура

### Многослойная архитектура

Проект следует принципам Clean Architecture с четким разделением ответственности:

```
┌─────────────────────────────────────┐
│   Presentation Layer (CLI)          │  ← CLI интерфейс (click)
├─────────────────────────────────────┤
│   Application Layer (Orchestration)   │  ← ReviewService, MRReviewService
├─────────────────────────────────────┤
│   Domain Layer (Business Logic)      │  ← Models, Validators, Prompts
├─────────────────────────────────────┤
│   Infrastructure Layer (External)    │  ← GitLab, LLM, Config, HTTP
└─────────────────────────────────────┘
```

### Структура проекта

```
luminary/
├── src/luminary/
│   ├── domain/              # Бизнес-логика
│   │   ├── models/          # FileChange, Comment, ReviewResult
│   │   ├── prompts/         # ReviewPromptBuilder, ValidationPromptBuilder
│   │   └── validators/      # CommentValidator
│   ├── application/         # Оркестрация
│   │   ├── review_service.py      # Ревью одного файла
│   │   └── mr_review_service.py    # Ревью всего MR
│   ├── infrastructure/      # Внешние интеграции
│   │   ├── llm/             # LLM провайдеры (OpenAI, OpenRouter, DeepSeek, VLLM, Mock)
│   │   ├── gitlab/          # GitLabClient
│   │   ├── config/          # ConfigManager
│   │   ├── http_client.py   # HTTP клиент с retry
│   │   ├── diff_parser.py   # Парсинг diff
│   │   └── file_filter.py   # Фильтрация файлов
│   └── cli.py               # CLI интерфейс
├── tests/                   # Тесты
├── docs/                    # Документация
│   └── ADR/                 # Архитектурные решения
└── examples/                # Примеры конфигураций
```

## 🔧 Ключевые компоненты

### 1. LLM Провайдеры

**Архитектура:** Strategy + Factory паттерн

- **Базовый класс:** `LLMProvider` (ABC)
- **Провайдеры:**
  - `OpenRouterProvider` — OpenRouter API (приоритет #1)
  - `OpenAIProvider` — OpenAI API
  - `DeepSeekProvider` — DeepSeek API
  - `VLLMProvider` — локальный OpenAI-compatible сервер
  - `MockLLMProvider` — для тестирования

**Особенность:** Большинство провайдеров наследуются от `OpenAICompatibleChatProvider`, что устраняет дублирование кода для OpenAI-совместимых API.

### 2. Обработка файлов

**Поток обработки:**
```
MR → Парсинг diff → Фильтрация → Последовательная обработка → Валидация → Отправка
```

- **Парсинг:** Поддержка unified diff формата и обычных файлов
- **Фильтрация:** Игнорирование бинарных файлов и паттернов (glob)
- **Обработка:** Последовательная (с возможностью миграции на параллельную)
- **Chunking:** Для файлов выше лимита контекста (с перекрытием)

### 3. Валидация комментариев

- **Валидатор:** `CommentValidator` с LLM-провайдером
- **Критерии:** Релевантность, полезность, не избыточность
- **Порог:** Настраиваемый (по умолчанию 0.7)
- **Обработка:** Отклонение невалидных с логированием

### 4. Конфигурация

**Иерархия приоритетов:**
1. Дефолтные значения
2. `.ai-reviewer.yml` файл
3. Environment variables
4. CLI аргументы

**Основные секции:**
- `llm` — настройки LLM провайдера
- `validator` — настройки валидации
- `gitlab` — настройки GitLab
- `ignore` — паттерны игнорирования
- `limits` — лимиты обработки
- `comments` — режимы комментариев
- `retry` — настройки retry логики
- `prompts` — кастомные промпты

### 5. GitLab интеграция

- **Клиент:** `GitLabClient` на базе `python-gitlab`
- **Функции:**
  - Получение данных MR
  - Парсинг diff в структурированный формат
  - Отправка inline и summary комментариев
  - Retry логика для API вызовов

## ⚠️ Известные проблемы и рекомендации по улучшению

### 1. Дублирование retry логики ✅

**Статус:** ✅ Решено

**Решение:**
- Retry логика унифицирована через библиотеку `tenacity`
- Создан единый модуль `infrastructure/retry.py` с утилитами для retry
- `http_client.py` и `GitLabClient` используют одинаковую логику retry
- Поддержка экспоненциального backoff с jitter для всех API вызовов
- Retry на основе типов исключений (HTTPError, GitlabError)

**Реализация:**
```python
from luminary.infrastructure.retry import create_retry_decorator

# Единая функция для создания retry декоратора
decorator = create_retry_decorator(retry_config, retry_condition, before_sleep)
```

### 2. Валидация конфигурации

**Статус:** ✅ Решено

**Решение:**
- Валидация конфигурации реализована через `pydantic>=2.0`
- Созданы Pydantic модели для всех секций конфига в `domain/config/`:
  - `LLMConfig` - настройки LLM провайдера
  - `ValidatorConfig` - настройки валидации комментариев
  - `GitLabConfig` - настройки GitLab интеграции
  - `IgnoreConfig` - паттерны игнорирования файлов
  - `LimitsConfig` - лимиты обработки
  - `CommentsConfig` - режимы комментариев
  - `PromptsConfig` - кастомные промпты
  - `RetryConfig` - настройки retry логики
  - `AppConfig` - главная модель, объединяющая все секции
- Валидация при загрузке конфига (fail-fast подход)
- Улучшенные сообщения об ошибках
- Обратная совместимость сохранена

**Реализация:**
```python
from luminary.domain.config import AppConfig
from luminary.infrastructure.config import ConfigManager

# Автоматическая валидация при загрузке
manager = ConfigManager()  # Raises ConfigurationError if invalid
config = manager.config  # AppConfig (Pydantic model)

# Типизированный доступ
llm_config: LLMConfig = manager.get_llm_config()
print(llm_config.provider)  # Type-safe, IDE autocomplete
```

**Преимущества:**
- Fail-fast: ошибки обнаруживаются при запуске
- Type Safety: полная типизация, автокомплит в IDE
- Валидация: автоматическая проверка типов, диапазонов, форматов
- Документация: схемы служат документацией
