# 🗺️ Roadmap разработки Luminary

## Обзор

Этот документ описывает план дальнейшего развития Luminary после завершения MVP (v0.1.0).

**Текущий статус:** MVP завершен, этапы 1.1 и 1.2 выполнены ✅  
**Версия:** 0.1.0

---

## ✅ Завершенные этапы (MVP)

### MVP включает:

- ✅ Полный цикл: CLI → GitLab MR → анализ → комментарии
- ✅ Поддержка LLM провайдеров: OpenRouter, OpenAI, DeepSeek, VLLM, Mock
- ✅ Валидация комментариев перед отправкой
- ✅ Работает в GitLab CI через Docker
- ✅ Гибкая конфигурация через YAML и CLI
- ✅ Обработка больших файлов (chunking)
- ✅ Режимы комментариев (inline, summary, both)
- ✅ Фильтрация файлов и retry логика
- ✅ Полная документация и тесты

---

## 🚀 Будущие улучшения

### Этап 1: Рефакторинг и оптимизация кода

**Цель:** Улучшить качество кода, устранить дублирование и технический долг.

#### 1.1. Унификация retry логики ✅

**Проблема:** Retry логика дублировалась в `http_client.py` и `GitLabClient._retry_api_call()`

**Статус:** ✅ Завершено

**Реализовано:**
- [x] Интегрирована библиотека `tenacity` (>=8.2.0)
- [x] Создан единый модуль `retry.py` с унифицированными утилитами
- [x] Рефакторинг `http_client.py` для использования tenacity
- [x] Рефакторинг `GitLabClient` для использования tenacity
- [x] Добавлен retry на основе типов исключений
- [x] Поддержка jitter и настраиваемого backoff_multiplier в GitLabClient
- [x] Обновлены все тесты

**Результат:** Устранено дублирование кода, retry логика унифицирована через библиотеку tenacity с сохранением всей функциональности.

#### 1.2. Валидация конфигурации ✅

**Проблема:** `ConfigManager` использует простой merge словарей без валидации схемы

**Реализовано:**
- [x] Интегрирована `pydantic>=2.0` для валидации конфигурации
- [x] Определены Pydantic схемы для всех секций конфига в `domain/config/`
- [x] Валидация при загрузке конфига (fail-fast подход)
- [x] Улучшенные сообщения об ошибках конфигурации
- [x] Обновлены и добавлены тесты (30 новых тестов валидации)
- [x] Обратная совместимость сохранена

**Результат:** Строгая типизированная валидация конфигурации с ранним обнаружением ошибок. Все секции конфига представлены отдельными Pydantic моделями с валидацией типов, диапазонов и форматов.

#### 1.3. Улучшение промптов и шаблонизации

**Текущее состояние:** Используются f-strings для шаблонизации промптов

**Задачи:**
- [ ] Рассмотреть миграцию на `Jinja2` для более сложных шаблонов
- [ ] Сохранить обратную совместимость с простыми промптами
- [ ] Добавить поддержку условной логики, циклов, фильтров в промптах
- [ ] Улучшить дефолтные промпты на основе статистики использования
- [ ] Добавить контекстную информацию (язык, фреймворк) в промпты

**Оценка:** 2-3 дня

#### 1.4. Обработка ошибок

**Задачи:**
- [ ] Создать иерархию кастомных исключений (`LuminaryError`, `ConfigError`, `LLMProviderError`, `GitLabError`)
- [ ] Более понятные сообщения об ошибках для пользователей
- [ ] Логирование с контекстом
- [ ] Graceful degradation при частичных сбоях
- [ ] Обновить документацию по обработке ошибок

**Оценка:** 2-3 дня

**Итого этап 1:** 9-13 дней

### Этап 2: Производительность и масштабируемость

**Цель:** Улучшить производительность и способность обрабатывать большие MR.

#### 2.1. Параллельная обработка файлов

**Текущее состояние:** Обработка файлов последовательная

**Задачи:**
- [ ] Добавить пул потоков/процессов на уровне Application Layer
- [ ] Сохранить интерфейсы Domain Layer без изменений
- [ ] Использовать `concurrent.futures` или `asyncio`
- [ ] Настраиваемое количество воркеров через конфиг
- [ ] Обработка ошибок в параллельном контексте
- [ ] Тесты для параллельной обработки

**Оценка:** 4-5 дней

#### 2.2. Асинхронность

**Текущее состояние:** Все HTTP вызовы синхронные (`requests`)

**Задачи:**
- [ ] Миграция на `httpx` или `aiohttp` для асинхронных вызовов
- [ ] Domain Layer остается без изменений
- [ ] Добавить async/await в Application Layer
- [ ] Асинхронные вызовы к LLM провайдерам
- [ ] Асинхронные вызовы к GitLab API
- [ ] Обновить все провайдеры для поддержки async
- [ ] Обновить тесты

**Оценка:** 7-10 дней

#### 2.3. Кэширование анализа

**Цель:** Избежать повторного анализа неизмененных частей кода

**Задачи:**
- [ ] Определить стратегию кэширования (на основе хеша кода)
- [ ] Реализовать кэш для результатов анализа файлов
- [ ] Инвалидация кэша при изменении кода
- [ ] Опциональное кэширование (можно отключить)
- [ ] Хранение кэша (память, файловая система, Redis)
- [ ] Метрики эффективности кэширования

**Оценка:** 5-7 дней

**Итого этап 2:** 16-22 дня
